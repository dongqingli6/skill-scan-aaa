# [ㄱ] 기억 시각화 MVP 구현계획서

## 1. 목적

현재 `[ㄱ]`은 기억 저장과 recall은 가능하지만, 사용자가 기억 상태를 직접 확인하거나 탐색하기는 어렵습니다.
이번 MVP의 목적은 아래 세 가지를 먼저 해결하는 것입니다.

1. 지금 얼마나 기억이 쌓였는지 한눈에 볼 수 있게 한다.
2. 저장된 기억을 검색하고 상세 내용을 확인할 수 있게 한다.
3. 연상 기억 연결 구조를 "생각나무" 형태로 시각화한다.

범위는 기본 뼈대 기능으로 제한합니다.
운영 분석, 정리 추천, 병합, 삭제 추천 같은 고급 기능은 MVP 이후 단계로 미룹니다.

## 2. MVP 범위

이번 단계에서 구현할 화면은 4개입니다.

1. Dashboard
2. Memory Search
3. Memory Detail
4. Association Graph

이번 단계에서 구현할 API는 5개입니다.

1. `GET /api/status`
2. `GET /api/memories`
3. `GET /api/memories/{node_id}`
4. `GET /api/graph/{node_id}`
5. `GET /api/search`

이번 단계에서는 수정 기능을 넣지 않습니다.
즉, 화면은 기본적으로 읽기 전용입니다.

## 3. 사용자 가치

이 MVP가 먼저 필요한 이유는 다음과 같습니다.

1. 기억이 실제로 저장되고 있는지 사용자가 직접 확인할 수 있습니다.
2. 어떤 기억이 어떤 기억과 연결되어 있는지 눈으로 볼 수 있습니다.
3. recall 품질 문제를 사용자가 감으로만 느끼지 않고, 저장 구조를 보면서 검토할 수 있습니다.
4. 이후 고급 기능을 넣기 전에 운영 가시성을 먼저 확보할 수 있습니다.

## 4. 화면 구성

### 4.1 Dashboard

가장 먼저 보여줄 요약 화면입니다.

표시 항목:

- 전체 기억 수
- 최근 24시간 저장 수
- 최근 7일 저장 수
- 마지막 저장 시각
- 전체 연결 수
- 평균 연결 수
- 최근 저장된 기억 10개

이 화면의 목적은 "시스템이 살아 있고 기억이 쌓이고 있다"를 바로 확인하게 하는 것입니다.

### 4.2 Memory Search

저장된 기억을 직접 찾는 화면입니다.

필수 기능:

- 키워드 검색
- 날짜 역순 정렬
- 최근 저장 순 기본 정렬
- 검색 결과 목록
- 각 결과의 짧은 미리보기

검색 결과 항목:

- `node_id`
- `timestamp`
- `text` 일부
- 연결 수

초기 MVP에서는 벡터 검색 UI까지는 넣지 않고, 일반 텍스트 검색과 목록 탐색부터 제공합니다.
벡터 기반 검색 결과 비교 화면은 후속 단계에서 고려합니다.

### 4.3 Memory Detail

기억 하나를 눌렀을 때 나오는 상세 화면입니다.

표시 항목:

- 전체 텍스트
- 저장 시각
- 직접 연결된 기억 목록
- 각 연결의 가중치
- 최근성 기준 관련 정보

이 화면은 "왜 이 기억이 중요한지"와 "어디로 연결되는지"를 확인하는 중심 화면입니다.

### 4.4 Association Graph

선택한 기억을 중심으로 연결 구조를 보여주는 시각화 화면입니다.

표시 규칙:

- 중앙 노드: 현재 선택한 기억
- 1-hop 연결: 직접 연결된 기억
- 2-hop 연결: 강한 연결만 제한적으로 확장
- 엣지 굵기: 가중치
- 노드 색상: 최근성
- 노드 클릭: Memory Detail로 이동

이 화면의 목적은 "생각나무"를 직관적으로 보여주는 것입니다.

## 5. 백엔드 설계

현재 저장 구조는 이미 대부분 준비되어 있습니다.

- `nodes`
  - `id`
  - `text`
  - `embedding`
  - `timestamp`
- `edges`
  - `source_id`
  - `target_id`
  - `weight`

즉, MVP에 필요한 핵심 데이터는 현재 DB로 충분합니다.
추가 테이블 없이도 1차 시각화는 가능합니다.

다만 UI에서 쓰기 좋도록 서비스 계층에 읽기 전용 메서드를 추가하는 것이 필요합니다.

권장 추가 메서드:

1. `list_memories(limit, offset, q)`
2. `get_memory(node_id)`
3. `get_connected_memories(node_id, limit)`
4. `get_graph_subtree(node_id, depth, limit)`
5. `get_dashboard_stats()`

## 6. API 설계

### 6.1 `GET /api/status`

목적:

- 대시보드 상단 요약 데이터 제공

응답 예시:

```json
{
  "db_path": "memory.db",
  "node_count": 128,
  "edge_count": 241,
  "last_timestamp": "2026-04-15T08:10:00+00:00",
  "recent_24h_count": 12,
  "recent_7d_count": 57
}
```

### 6.2 `GET /api/memories`

목적:

- 최근 기억 목록
- 키워드 필터 결과 목록

쿼리 파라미터:

- `q`
- `limit`
- `offset`

### 6.3 `GET /api/memories/{node_id}`

목적:

- 기억 하나의 상세 정보 제공

응답 항목:

- 본문 텍스트
- 저장 시각
- 직접 연결된 기억 목록
- 각 연결 가중치

### 6.4 `GET /api/graph/{node_id}`

목적:

- 그래프 시각화용 서브트리 제공

쿼리 파라미터:

- `depth`
- `limit`

응답 구조:

- `nodes`
- `edges`

### 6.5 `GET /api/search`

목적:

- 검색 화면 전용 API

초기에는 `GET /api/memories?q=...` 와 역할이 겹칠 수 있으므로, 실제 구현에서는 하나로 합쳐도 됩니다.
단, UI 설계 분리를 위해 문서에서는 별도 항목으로 둡니다.

## 7. 프론트엔드 설계

MVP에서는 복잡한 프레임워크보다 유지보수가 쉬운 구성이 좋습니다.

권장안:

- 백엔드: `FastAPI`
- 프론트엔드: 서버 템플릿 + 가벼운 JS
- 그래프: `Cytoscape.js` 또는 `vis-network`

이유:

- 현재 프로젝트 성격상 로컬 도구에 가깝습니다.
- 관리자 화면처럼 빠르게 붙이는 것이 중요합니다.
- React SPA를 바로 올리는 것은 초기에 과합니다.

화면 구조:

- `/ui`
  - Dashboard
- `/ui/memories`
  - Search
- `/ui/memories/{node_id}`
  - Detail
- `/ui/graph/{node_id}`
  - Association Graph

## 9. MVP 완료 기준

아래가 되면 MVP 완료로 봅니다.

1. 브라우저에서 로컬 UI를 열 수 있다.
2. 전체 기억 수와 최근 저장 상태를 볼 수 있다.
3. 저장된 기억을 텍스트로 검색할 수 있다.
4. 기억 하나를 눌러 상세 내용과 연결 목록을 볼 수 있다.
5. 선택한 기억 기준 생각나무 그래프를 볼 수 있다.

## 10. 제외 범위

이번 단계에서는 아래를 구현하지 않습니다.

- 기억 수정
- 기억 삭제
- 병합 추천
- recall/reinforce 이력 분석
- 히트맵
- 태그
- 품질 점수
- 이상 연결 탐지
- 벡터 검색 결과와 키워드 검색 비교 UI

이 기능들은 MVP가 안정화된 뒤 2단계에서 검토합니다.

## 11. 권장 후속 단계

MVP 다음 단계에서 가장 가치가 큰 기능은 아래 순서입니다.

1. Recall Trace
2. Memory Quality 진단
3. Memory Hygiene

이 셋은 시각화 화면이 실제 운영 도구가 되게 만드는 기능입니다.

## 12. 최종 제안

이번에는 "볼 수 있는 기본 뼈대"까지만 갑니다.

정리하면 구현 우선순위는 아래와 같습니다.

1. Dashboard
2. Search
3. Detail
4. Association Graph
5. 읽기 전용 API

이 범위가 가장 작고, 가장 빨리 가치가 나며, 이후 고급 기능으로 확장하기도 쉽습니다.
