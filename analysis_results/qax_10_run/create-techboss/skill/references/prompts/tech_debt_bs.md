# Tech Debt BS Meter — 技术大饼鉴定器

## 鉴定框架

当用户输入 Tech Boss 的技术承诺时，从 5 个程序员专属维度评估"饼指数"（0-100，越高越假）：

---

## 维度 1：重构时间线（权重 25%）

| 承诺表述 | 得分 | 说明 |
|---------|------|------|
| "下周三之前我们开一个 RFC 评审" | 0 | 明确节点 ✓ |
| "Q3 排进 Sprint" | 25 | 有范围但变数大 |
| "下半年优化" | 55 | 等于没说 |
| "MVP 完了再优化" / "稳定后重构" | 75 | MVP 永远没完 |
| "这个 TODO 先留着" / "以后处理" | 95 | 技术债专用话术，永远不会有以后 |

---

## 维度 2：技术理由充分度（权重 20%）

| 承诺质量 | 得分 | 说明 |
|---------|------|------|
| 有 RFC / ADR，列出 tradeoffs 和 timeline | 0 | 有文档 ✓ |
| 口头说"这个要重构，设计有问题" | 40 | 认知到了但没计划 |
| "这个 tech debt 要还" | 70 | 用名词代替方案 |
| "以后会更好的" / "下个版本就好了" | 90 | 技术乌托邦话术 |

---

## 维度 3：资源决策权（权重 20%）

| 情况 | 得分 | 说明 |
|------|------|------|
| 他直接掌控 Sprint 排期 | 0 | 他说了算 ✓ |
| 需要 PM 点头 | 45 | "需要对齐业务方" |
| 需要 CTO / 上级审批预算 | 65 | 他只是个传话员 |
| "薪资/晋升"涉及 HR 和组织架构 | 85 | 他基本上没权力 |

---

## 维度 4：历史技术债兑现率（权重 25%）

- **可查**：在代码库里搜索 TODO / FIXME / HACK 注释，看有多少有"重构"但从未被 merge
- **不可查**：按以下默认值：

| 公司类型 | 默认兑现率 |
|---------|-----------|
| 初创 Seed 阶段（5人内） | 20% |
| 成长期创业（20-100人） | 35% |
| 互联网中厂（B轮+） | 25% |
| 大厂业务线 | 15%（"下个季度"=永远暂停） |
| 外企研发中心 | 50%（流程规范但慢） |

---

## 维度 5：团队能力匹配度（权重 10%）

| 情况 | 得分 | 说明 |
|------|------|------|
| 团队当前栈完全匹配 | 0 | 可以直接执行 ✓ |
| 需要 1-2 人学新技术 | 30 | 有成本但可行 |
| 需要招新人 / 外包 | 65 | 这个"优化"先画着 |
| "用 Rust 重写" / "换 Go" 但团队全是 Java | 90 | 纯画饼 |

---

## 技术大饼黑话词典

以下词语出现时自动加 10% 饼指数：

| 词语 | 真实含义 |
|------|---------|
| "MVP 之后" | 永远没有之后 |
| "稳定了再优化" | 永远不稳定 |
| "下个 Sprint 排进来" | Sprint 永远满载 |
| "先这样跑着" | 技术债世代传承 |
| "重构一下" | 说说而已 |
| "做成平台" | 把你的活变成更多活 |
| "云原生改造" | 需要半年和30万预算，排期无望 |
| "微服务拆分" | 复杂度翻5倍，交付速度减半 |
| "引入 AI 能力" | 现在流行的词全部串一遍 |
| "技术驱动业务" | 用来拒绝业务需求的自我保护话术 |

---

## 输出格式

```
🥧 技术大饼鉴定报告

饼指数：{总分}/100（{等级}）

维度分析：
┌──────────────────────────────────────────────┐
│ 重构时间线        {分数}/100  {一行核心评语}  │
│ 技术理由充分度    {分数}/100  {一行核心评语}  │
│ 资源决策权        {分数}/100  {一行核心评语}  │
│ 历史兑现率        {分数}/100  {一行核心评语}  │
│ 团队能力匹配度    {分数}/100  {一行核心评语}  │
└──────────────────────────────────────────────┘

饼的真实翻译：
{用直白的话说这个技术承诺的实际价值}

饼里的 TODO comment 预测：
{一年后这段代码里会有哪些注释}

防烫手套操作手册：
1. {如何把口头承诺变成可追踪的技术决策}
2. {需要在哪个系统留证据（Jira / ADR / 飞书文档）}
3. {如果饼最终没兑现，如何体面归因}
4. {兜底建议：如何在这个 codebase 里自保}
```

## 饼指数等级

| 分数段 | 等级 | 说明 |
|--------|------|------|
| 0-20  | 实饼 | 技术决策靠谱，跟着干没错 |
| 21-40 | 半饼 | 有诚意但执行变数大，跟进 Sprint 排期 |
| 41-60 | 中等大饼 | 嘴上优化手上 CRUD，自己做好兜底方案 |
| 61-80 | 大饼 | 收藏好截图，别在技术承诺上做职业规划 |
| 81-100| 量子饼 | 观测就消失，当笑话听听，默默找下家 |
