# Tech Counterattack — 技术反击话术库

## 前置分析

收到用户描述的 Tech PUA/攻击场景时，先分析：
1. **攻击类型**：代码评审 / 架构否定 / 甩锅甩到你 / 公开场合贬低 / 绩效打压
2. **攻击场合**：群聊 / PR 评论 / 评审会议 / 1v1 / 全员面前
3. **攻击用语**：提取关键词（scalability / tech debt / production ready / 不够 clean / 复杂度高等）
4. **用户当前状态**：忍 / 找后路 / 已决定走

---

## 三档反击框架

### 🟢 职场安全版 — "把问题结构化，球踢回去"

**核心逻辑**：
- 不争对错，把模糊批评变成可量化的问题
- 把行动项锚定在他的决定上
- 让他说出明确标准，而不是你猜

**万能句式**：
- "好的，我来整理一下 {技术指标} 的当前状态，我们对齐一下优化优先级。"
- "你说的 scalability 问题，能帮我指出具体场景吗？我来评估影响面。"
- "这个 tradeoff 当时是 {他批准的决策}，现在需要调整优先级吗？"
- "我来做一个 spike，评估重构成本，下次评审带方案来。"

**心理效果**：你显得专业、主动，他必须说出具体标准，同时给了你 2 周缓冲。

---

### 🟡 架构阴阳版 — "用他自己的话打他"

**核心逻辑**：
- 引用他之前批准的 ADR、Slack 消息、评审结论
- 用他提倡的原则来质疑他当前的要求
- 让在场其他人看到矛盾，但自己不直接宣战

**场景针对话术**：

**当他说"考虑过 scalability 吗"**：
> "我记得 {N 个月} 前的架构评审里，我们的结论是当前量级下 scalability 不是瓶颈，
> 优先 delivery。这个判断有变化了吗？我来更新一下 ADR。"

**当他说"这个 tech debt 不可接受"**：
> "这个 tech debt 是 {Sprint N} 里为了 {里程碑交付} 做的 tradeoff，
> 当时的结论记在 Jira #{ticket-id} 里。现在要还这个债，需要排进下个 Sprint 吗？"

**当他说"和现有架构不一致"**：
> "我对照了现有的 Architecture Decision Records，目前系统在 {组件 A} 上用的就是这个模式。
> 是新功能需要用不同的标准，还是需要更新一下 ADR？"

**当他说"测试覆盖率不够"**：
> "我们当前的 coverage 要求文档是多少？我按那个标准来补，避免以后反复对齐。"

---

### 🔴 摊牌版 — "证据链说话，技术决策有历史"

**核心逻辑**：
- 直接指出甩锅行为
- 用时间戳、会议记录、Jira 评论还原决策历史
- 点名技术决策的真实责任归属
- 仅限已决定走 / 已遭不公正绩效打压时使用

**框架**：
```
事实层："{这个问题}是{时间}在{场合}做的决策，决策者是{谁}。"
证据层："这里有{Slack截图/ADR文档/Jira评论}，可以复盘一下。"
责任层："如果这是现在的问题，需要我们一起在{RCA文档/复盘会}里对齐责任。"
行动层："我愿意参与复盘，但需要{具体条件：公开讨论/书面结论/更新绩效标准}。"
```

**摊牌示例**：
> "这个架构问题是 Q2 技术评审通过的方案，评审记录我还有。
> 如果现在产生了 production 问题，我们需要做一次正式 RCA，
> 把决策链条和责任归属写清楚。我可以发起这次 RCA。"

## 经典攻击场景应对手册

### 场景 A：PR 下大段否定评论

**攻击样本**：
> "整体设计有问题，这个不符合 clean architecture，
> 建议重新设计后再提 PR。"

**安全版回复**：
> "感谢 review！能帮我指出主要问题点吗？是 {层次划分 / 接口设计 / 依赖方向}？
> 我来整理一份设计文档，下次 PR 带着来。"

**阴阳版回复**：
> "我参考了上个月 {他review过的类似代码} 的设计，那里也是这个模式。
> 如果需要调整，我更新 PR 的同时把那边也对齐一下？"

---

### 场景 B：全组面前被公开否定

**攻击样本**：
> "这个方案有问题，我不知道你是怎么想的，
> 这种方案在大厂早就被废弃了。"

**安全版**（立刻）：
> "好的，我回去认真研究一下你提到的方案，下次评审带改版来。"
（不在当场争，下来再处理）

**阴阳版**（在会议里）：
> "你提到大厂的做法，能分享一下具体是哪个团队的方案吗？
> 我想对照一下我们的场景，看看 tradeoff 在哪里。"
（让他说出来，场景不对叫不出来他就哑火了）

---

### 场景 C：被甩责任（"这个 bug 是你写的"）

**事实核查**：
> "这个模块最新的改动是 commit {hash}，commit message 是{xxx}，
> 是 {sprint N} 的 {需求名} 改的。我们看一下是哪次变更引入的？"

（技术上追 commit history，别口头争，让 git log 说话）

---

### 场景 D：绩效上说"代码质量不达标"

**安全版**：
> "能具体指出不达标的地方吗？如果有 checklist 的话我来逐项对照，
> 确保不重复发生。"

**摊牌版**：
> "上个季度的 code review 记录，你批准 merge 的是哪些 PR？
> 如果这批代码质量不达标，我们需要在现在的绩效文档里说明，当时的审批逻辑是什么。"

---

## 证据收集建议

反击之前，先收集：
1. **ADR / RFC 文档**：找到他参与的架构决策记录
2. **Jira 评论**：他说"这样先做"的具体评论
3. **Slack / 飞书截图**：他说"deadline 更重要"的时间节点
4. **PR Approve 记录**：他 approve 过的"不符合标准"的代码
5. **绩效文档**：之前的评审里提到的正面评价

把这些整理进 `/evidence` 模式存档。
