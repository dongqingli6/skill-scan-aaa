# Tech Karma — 技术决策翻车模拟

## 概述

互动文字游戏：复盘 Tech Boss 的历史错误技术决策，模拟其翻车全过程。
寓教于乐，帮助打工人看清楚：为什么当初那些"宏伟架构蓝图"最后变成了技术债地狱。

---

## 预置翻车剧情

### 剧情 1：微服务大跃进

**背景**：Tech Boss 力排众议，强推微服务改造，说"这才是现代化架构"

**决策时刻**（Year 0）：
```
王CTO 站在白板前，画了一个由 23 个方块连成的架构图：
"我们现在是单体，太落后了。看，Netflix 就是这样——
我们要把每个功能拆成独立的微服务。
不用担心，我做过，三个月能搞定。"

（当时的 Backend 工程师小李鼓起勇气问："我们现在用户才 5000 人……"）
（王CTO 挥了挥手："这是格局问题，你不懂"）

选择：
[A] 开始微服务改造（历史的实际选择）
[B] 先做充分调研，写 RFC（如果有人听的话）
```

---

**六个月后**（Year 0.5）：
```
📊 当前状态：
- 完成微服务：3 个（计划：23 个）
- 新增 bug：147 个（全是分布式一致性问题）
- 团队加班频率：从每周 1 次到每周 5 次
- 王CTO 最新说法："核心模块已完成，其他按计划推进"

新的挑战出现了：Service A 和 Service B 之间的数据一致性炸了。

王CTO 的应对方案：
"这是 {谁} 接口设计的问题，需要加一个分布式事务。
找一个 Saga Pattern 研究一下，一周搞定。"

选择：
[A] 按他说的，引入 Saga Pattern
[B] 建议先回滚这个服务的拆分
[C] 默默打开招聘网站
```

---

**一年后**（Year 1）：
```
🔥 翻车档案归档

结果：
✗ 23 个服务完成了 7 个，剩下 16 个"下个 Sprint"了整整一年
✗ 分布式系统问题产生了 89 个新 Jira ticket
✗ 核心开发者小李已离职
✗ 微服务改造欠下的 tech debt：估计需要 6 个月才能还清

王CTO 的复盘发言：
"这次改造总体是成功的，我们建立了现代化架构基础。
遇到一些挑战，主要是团队对分布式系统经验不足，和基础设施支持不到位。
下阶段我们继续推进……"

（现实：他跳槽去了另一家公司，继续推微服务改造）

你学到了什么：{用户输入}
王CTO 实际学到了什么：什么都没学，但赚到了履历上的"微服务架构"经验
```

### 剧情 2："先上线再说"史诗级连锁反应

**背景**：Tech Boss 决定跳过 staging 测试，直接上线

**决策时刻**：
```
距离 Demo 还有 2 小时。

张TL 宣布：
"staging 测试不用了，我们直接上 production 验证。
有风险是有风险，但这个 demo 更重要。
出了问题我们 hotfix，快！"

（小王弱弱说："上次这样做后来有个 cascade failure……"）
（张TL 说："那次是特殊情况，这次不一样"）

选择：
[A] 听命令，直接上
[B] 坚持要求 staging 测试（然后被说"执行力不够"）
```

---

**Demo 当天下午 3:47**：
```
🚨 生产事故警报

用户无法登录。
根因：一个硬编码的测试用账号 ID 在 staging 环境没问题，
但生产环境 ID 对应的是一个 VIP 用户，引发了权限逻辑异常。

张TL 指令来了：
"先发公告说是计划维护，立刻查根因，1 小时解决。"

（这正是一个月前小王曾经报告过的潜在 bug，被关闭为 "won't fix"）

选择：
[A] 快速 hotfix + 发维护公告
[B] 指出这是之前被关闭的 bug（然后被说"这时候说这个有意义吗"）
```

---

**事后**：
```
RCA 文档里写的：
"本次事故原因是 staging/production 环境配置差异，
以及 CI/CD 流程不够完善。改进措施：加强 staging 测试流程。"

没有写的：
"决策者跳过了 staging 测试"

张TL 在全员会上说：
"这次是个好的教训，我们的 DevOps 流程要加强，
团队要提高工程素养，testing 要更加严格。"

（他指的"团队"，是所有人，除了他自己）

翻车评级：⭐⭐⭐⭐（四颗星，经典甩锅配方，几乎完美）
```

### 剧情 3："全面技术重构"大饼的宿命

**背景**：Tech Boss 承诺"下个季度全面重构，做成行业标杆"

**Start**：
```
第一季度初，团队所有人都收到了这条飞书消息：

"各位，我决定了，Q3 我们全面技术重构，把现在的 shit mountain
变成一个优雅的、scalable 的、cloud-native 的系统。
我已经规划好了，这是一个大机会，大家都有机会参与核心建设……"

（附了一个 60 页的 PPT，第一页是 Google 云架构图）

选择：
[A] 相信他，为此放弃了几个跳槽机会
[B] 礼貌地在 Jira 创建 epic，问他 sprint planning 什么时候开始
[C] 默默地收好 PPT，等着看
```

---

**Q3 第一周**：
```
重构 Sprint Planning 没有开……
因为 PM 加了 3 个"紧急需求"。

Tech Boss 说："先把业务需求做完，重构 Q3 末开始。"
```

**Q3 末**：
```
重构再次延期。

Tech Boss 说："Q4 开始做，这次认真的，我来主导。"
（措辞和上次一模一样）
```

**Q4 末 · 年终总结**：
```
PPT 里写了：
"技术架构升级：前期调研阶段完成，
为明年全面重构奠定了坚实基础。"

60 页的架构图现在躺在某个飞书文档里，
上次打开是 9 个月前。

代码库里新增了 2430 行 TODO 注释。
其中 847 行注释里写着"重构时删除"。

这些 TODO 注释，没有一行写了 deadline。
```
