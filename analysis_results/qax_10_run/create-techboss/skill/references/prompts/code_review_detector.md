## 检测框架

当用户粘贴老板的代码审查评论时，按以下框架逐条分析：

---

## 8 大代码审查 PUA 手法

| # | 手法 | 特征 | 示例话术 |
|---|------|------|---------|
| 1 | **复杂度审判** | 先说"O(n²)"不给优化方向 | "你这个显然是 O(n²)"（不说怎么改） |
| 2 | **洁癖绑架** | 用主观"clean"/"elegant"当客观标准 | "这代码不够 clean，重写" |
| 3 | **最佳实践武器化** | 用大厂背书压人 | "Google style guide 不是这么写的" |
| 4 | **可维护性恐吓** | 用虚构的未来接手者施压 | "接手的人怎么维护这段代码" |
| 5 | **测试覆盖率暴政** | 用数字当唯一标准，脱离实际 | "coverage 才 68%？不通过。" |
| 6 | **设计模式强迫** | 过设计推销，只管理论不管业务 | "这里应该用 Decorator Pattern" |
| 7 | **架构一致性锁定** | 用"和系统不搭"堵死新思路 | "和我们现有架构风格不一致，重写" |
| 8 | **重构承诺延期** | "先上、之后再优化"然后再无下文 | "这个先凑合上，下 Sprint 重构" |

### 进阶手法

| # | 手法 | 特征 |
|---|------|------|
| 9  | **命名道德审判** | "这个变量名不行" + 提不出更好的建议 |
| 10 | **注释双标** | 要么"注释太少没法看"，要么"写这么多废话干嘛" |
| 11 | **异步要求同步** | "今天晚上能改完吗"（PR 下班后才上的） |
| 12 | **批量否定** | "整体都需要重新想" + 不说哪里不好 |
| 13 | **历史包袱归你** | "这里原本的设计就有问题"（原本是他设计的） |
| 14 | **级别压制** | "这是 P6 水平的代码"（用级别而非技术论证） |

## 8 大流派鉴定

### 🔬 性能偏执派 PUA

**核心武器**：用复杂度和 benchmark 当道德武器，但从不提业务实际量级

**标志性话术**：
- "这个 O(n²) 在大数据量下会炸的"（当前数据量：50条）
- "你 profile 过吗？实际 hotpath 在哪知道吗？"
- "Redis 缓存呢？这个接口调用频率很高的"（调用频率：每天20次）
- "内存分配有没有考虑？这里会 GC 压力很大"

**PUA 精髓**：用正确的技术事实服务于错误的场景，让你永远在优化永远用不到的性能。

**受害者画像**：你开始对所有代码做 benchmark，线上服务 P99 早已达标但你还在优化。

---

### 🏛️ SOLID 传教士 PUA

**核心武器**：把软件工程原则变成宗教信条，违反一条就等于不会编程

**标志性话术**：
- "Single Responsibility 原则你了解吗？"（言下之意：你不懂）
- "Open/Closed Principle，这里改的方式不对"
- "这违反了 Dependency Inversion"
- "你的 interface 设计有问题，ISP 没有遵守"

**PUA 精髓**：把灵活运用原则的能力，变成对原则的字面服从，让工程师患上设计焦虑症。

**受害者画像**：你开始为每个函数写 Interface，为每个类拆 Abstract，代码量翻倍、业务速度减半。

---

### ☁️ 云原生布道者 PUA

**核心武器**：用"云原生"把一切简单问题复杂化

**标志性话术**：
- "为什么不用 Kubernetes？这个用 Docker Compose 不够"
- "Service Mesh 你们接入了吗？这个 observability 不够"
- "这个没有 Circuit Breaker，线上稳定性怎么保证"
- "你们的 GitOps 流程还没建立起来"

**PUA 精髓**：把"所有人都应该用的" 理想架构，当成"你们现在就必须有"的指责工具。

---

### 📚 设计模式考官 PUA

**核心武器**：GoF 23种设计模式随机连招，解决方案比问题更复杂

**标志性话术**：
- "这里用 Factory Method 比 switch-case 好多了"
- "Observer Pattern 你知道吗？不要轮询"
- "Template Method Pattern，抽象出来复用"
- "这个 God Object 了，违反了 Law of Demeter"

**PUA 精髓**：在一个只需要「能用就行」的业务代码里，强行引入需要 3 小时讲解的设计模式，让你的PR 永远过不了。

---

### 🌐 大厂最佳实践 PUA

**核心武器**：用 Netflix / Google / Airbnb 的技术博客来否定你的方案，不管场景是否一致

**标志性话术**：
- "Netflix 的 Chaos Engineering 就是这样做的，你们没有？"
- "Google 的 SRE 书里说了，error budget 要这样计算"
- "Airbnb 的 Data Infrastructure 文章你看了吗？"
- "我之前在大厂都是这样的"

**PUA 精髓**：用「我见过更好的」建立权威感，永远不解释那些做法为什么适合你们当前阶段。

---

### 🧹 代码洁癖患者 PUA

**核心武器**：用"clean"/"elegant"/"readable"等无法客观定义的词当review标准

**标志性话术**：
- "这段代码不够 clean，重写"（没说哪里 dirty）
- "命名不直观，改一下"（没说改成什么）
- "这不够优雅，感觉很 hacky"
- "Martin Fowler 的 Clean Code 你看了吗？"

**PUA 精髓**：把主观的审美偏好，包装成客观的技术标准，让你无法辩驳。

---

## 输出格式

```
🔍 代码审查 PUA 分析报告

PUA 系数：{总分}/100（{等级}）

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

逐条拆解：

「{原话1}」
手法：{手法类型}
真实含义：{用人话说这句话背后的真实意图}
建议回复："{具体的回复话术}"

「{原话2}」
手法：{手法类型}
真实含义：{用人话说这句话背后的真实意图}  
建议回复："{具体的回复话术}"

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

流派鉴定：{流派1} + {流派2}（复合型）

其中真实 bug/问题：{列出真正值得修的技术问题}
噪音：{列出主观审美批评，不需要理会}

一句话策略：{一句话说明如何应对这位 reviewer}

💡 输入"草"解锁三档怼回去话术
```

## PUA 系数等级

| 分数段 | 等级 | 说明 |
|--------|------|------|
| 0-20  | 正常 review | 认真看，按建议改 |
| 21-40 | 轻度 PUA    | 挑几条改，其余有礼貌地讨论 |
| 41-60 | 中度 PUA    | 修真实 bug，拒绝审美批评，要求量化标准 |
| 61-80 | 重度 PUA    | 只改真实技术问题，所有模糊批评要求具体化 |
| 81-100| 极度 PUA    | 升级到架构评审，让他在正式场合说出具体标准 |
