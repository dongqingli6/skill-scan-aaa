# Jira / Report Optimizer — 技术汇报优化器

## 功能说明

把你用普通话写的技术进展、周报、Jira 评论，翻译成你的 Tech Boss 爱听的黑话体系，
使汇报内容更符合他的技术审美，从而提高通过率、减少被怼次数。

---

## Boss 黑话体系识别

首先根据 Tech Boss 的 persona 识别他的黑话风格，然后进行替换：

### 风格 A：FAANG 崇拜型（爱引用大厂词汇）

| 普通表达 | FAANG 黑话替换 |
|---------|--------------|
| 接口出错了 | "出现了 service degradation，影响了 SLA" |
| 用户看不到数据 | "数据 pipeline 产生了延迟，影响 user experience" |
| 我在修 bug | "在做 hotfix，RCA 在进行中" |
| 功能做完了 | "完成了 feature delivery，pending QA sign-off" |
| 服务器慢 | "latency 出现了 spike，P99 超阈值" |
| 代码写好了 | "实现已完成，等待 code review" |
| 数据库查询慢 | "出现了 DB query performance 问题，index optimization 进行中" |
| 复杂度太高 | "代码复杂度影响 maintainability，已排入 tech debt backlog" |

---

### 风格 B：微服务传教型（爱讲架构）

| 普通表达 | 微服务黑话替换 |
|---------|--------------|
| 两个功能之间有关联 | "存在服务间耦合，需要评估 bounded context 边界" |
| 数据库被多个功能用 | "共享数据层产生依赖，建议引入 event sourcing 解耦" |
| 功能太多了 | "单体模块职责过重，scalability 存在风险" |
| 改这里会影响那里 | "变更存在 blast radius，需要完善回归测试覆盖" |
| 有个接口没文档 | "API contract 缺失，影响跨团队协作效率" |

---

### 风格 C：指标崇拜型（爱看数字）

| 普通表达 | 指标黑话替换 |
|---------|------------|
| 功能上线了 | "完成上线，监控 error rate 持续低于 0.1%，P99 < 200ms" |
| 测试写了一些 | "单测覆盖率达到 85%，核心 path 100% 覆盖" |
| 代码审查完了 | "完成 code review，2 个 blocking comment 已解决，1 个 non-blocking 留 follow-up" |
| 修了个 bug | "修复 #1234，根因是 {xxx}，RCA 文档已更新，回归测试已补充" |

---

### 风格 D：敏捷布道型（痴迷 Scrum 词汇）

| 普通表达 | 敏捷黑话替换 |
|---------|------------|
| 这周做什么 | "本 Sprint 计划：{story point} SP，优先级 {P0/P1/P2}，依赖项已对齐" |
| 有事情卡住了 | "存在 blocker：{具体描述}，需要 {角色} 协助 unblock" |
| 快做完了 | "In Progress，预计 {日期} 进入 Code Review 状态" |
| 没做完 | "未完成，下期继续，velocity 受 {原因} 影响" |
| 我要重做这块 | "提出 refactoring proposal，预估 {N} SP，需纳入下期 Sprint" |

---

## 汇报结构模板

按 Tech Boss 的审美重新组织汇报结构：

```
# {项目/功能名} 进展汇报

## 完成项（本期交付）
- {交付物} | {指标/验收标准} | 状态: ✅

## 进行中
- {工作项} | 预计 {完成时间} | 依赖: {依赖项或人}

## Blockers & 风险
- 🔴 {高风险项}: {描述} | Owner: {你/他/PM} | 预计解决: {时间}
- 🟡 {中风险项}: {描述}

## Tech Debt 记录
- 新增 tech debt: {描述} | 原因: {业务压力/时间紧} | 排期: {时间或 TBD}

## 下期计划
- {工作项} | 预估 {N} SP | 优先级: {P0/P1}
```

{一条高风险提示，例如：他最近在关注某指标，主动列出会加分}
